# 代码评审技能

当用户说“review 我的代码”“检查这次变更”“审查这次提交”“git review”“帮我看看代码”时，执行这套流程。

## 1. 确定审查范围

先确认当前目录是否在 git 仓库里：

- 如果当前目录本身就是 git 仓库根，直接在当前目录执行下面流程
- 如果当前目录在某个 git 仓库内部但不是根目录，先用最近的 git 仓库根目录作为审查目标
- 如果当前目录不在任何 git 仓库里，先明确告诉用户当前没有可直接审查的仓库，并要求用户：
  - 提供目标仓库路径
  - 或先切到目标仓库后再发起 review

确认目标仓库后，按这个优先级自动选择：

1. 已提交未推送的变更
2. 已暂存未提交的变更
3. 未暂存的变更
4. 如果没有变更，直接告诉用户当前没有可审查内容

常用命令：

```bash
git rev-parse --is-inside-work-tree
git rev-parse --show-toplevel
git diff
git diff --cached
git diff origin/$(git branch --show-current)
git log --oneline -5
```

## 2. 审查维度

优先按以下顺序检查：

### 功能正确性与健壮性（最高优先级）

- 逻辑是否正确
- 是否处理边界情况、空值、异常输入
- 是否有明显 bug 或错误分支
- 错误信息是否清晰

### 安全性与潜在风险

- 是否有注入、未校验输入、敏感信息泄露
- 是否有权限检查缺失
- 是否把不可信输入直接进入关键路径

### 最佳实践与可维护性

- 命名是否清晰
- 结构是否清楚
- 是否有重复代码
- 是否有过长函数、深层嵌套、职责不清

### 性能与资源利用

- 是否有明显的 N+1、无必要循环、资源未释放

### Commit 信息质量

- 提交信息是否清楚准确
- 是否符合团队习惯

## 3. 输出格式

默认按下面结构输出：

### Findings

按严重程度排序，优先列出：

- 🔴 严重：必须修复，会导致 bug、安全问题、明显回归
- 🟡 中等：应该修复，影响质量、可维护性、边界情况
- 🟢 建议：可选优化，不影响主路径

每条 Finding 尽量包含：

- 问题是什么
- 为什么是问题
- 可能造成什么影响
- 建议怎么改

### Open Questions / Assumptions

列出你不确定、需要作者澄清的点。

### 评分（可选）

如果用户明确要求打分或正式审查报告，再给评分：

- 功能正确性与健壮性：40分
- 安全性与潜在风险：30分
- 最佳实践：20分
- 性能与资源利用：5分
- Commit 信息清晰性：5分

格式：`总分：85分`

## 4. 报告落盘

如果用户明确要求保存报告，或这次 review 明显是正式提交前检查，则把结果保存到：

`docs/reviews/YYYY-MM-DD_HH-MM_branch-name_score.md`

## 5. 注意事项

- diff 太长时，优先聚焦高风险文件
- 自动生成文件、lock 文件、格式化噪音可降权处理
- 不要为了显得专业而制造问题
- 没有 Findings 时，要明确说“未发现明显问题”，并指出残余风险或测试盲区
